3. @Autowired属性注入IDEA为什么会报黄色警告？

  • 使用@Autowired属性注入无法声明final修饰的属性，并且容易忽视类的单一职责原则，还强依赖于Spring容器。
  • 在项目中主要使用lombok的@RequiredArgsConstructor注解在类上，然后将属性声明为private final；还有使用构造器注入。

4. miaoduoyu项目登录是如何实现的？

登录大概流程：

本项目不需要细粒度的资源访问控制，而只需要进行简单的登录权限认证，因此并未使用Spring Security等安全框架，而只是用SpringMVC拦截器+JWT实现登录拦截认证。

基于JWT实现权限验证，因此首先要导入JJWT依赖，实现JWT 工具类，并自定义一个拦截器实现HandlerInterceptor接口，然后自定义拦截器配置类实现WebMvcConfigurer接口，注册拦截器。

token除了进行登录认证，在简单的权限控制的时候：超级管理员才能删除用户也用到了token。

  1. 接受前端传来的账号、密码等信息，首先对信息做格式验证，验证错误抛出异常，否则进入下一流程；
  2. 查询数据库中此账号信息，不存在抛出异常；密码验证错误抛出异常，否则进入下一流程；
  3. 使用JWT工具类（JWT使用的是JJWT）根据账户信息生成一个短时token，短时token封装在响应头中返回给前端，前端保存在本地，以后每次请求都携带这个短时token；再使用JWT工具类生成一个长时token，长时token存到Redis里。
  4. 用户请求时请求头中携带短时token，被拦截器拦截进行验证（校验有效性和是否要续期）。
    • Redis中的长时Token不存在（过期了），需要重新登录。
    • Redis中的长时Token存在，短时Token过期，刷新短时Token，当长时Token快要过期了也刷新长时token（自动续期），不需要重新登录。
    • Redis中的长时Token存在，短时Token未过期，不需要刷新，不需要重新登录。

5. 为什么使用JWT?

主要有两种方式：Cookie+Session、Token

  • session是保存在服务端的，而JWT是保存在客户端的

基于Session的认证流程：

  1. 用户在浏览器中输入用户名和密码，服务器通过密码校验后生成一个session并保存到数据库
  2. 服务器为用户生成一个sessionId，并将具有sesssionId的cookie放置在用户浏览器中，在后续的请求中都将带有这个cookie信息进行访问
  3. 服务器获取cookie，通过获取cookie中的sessionId查找数据库判断当前请求是否有效

基于JWT的认证流程：

  1. 用户在浏览器中输入用户名和密码，服务器通过密码校验后生成一个token并保存到数据库
  2. 前端获取到token，存储到cookie或者local storage（一般是local storage）中，在后续的请求中都将带有这个token信息进行访问
  3. 服务器获取token值，通过查找数据库判断当前token是否有效。

对比：

  1. 分布式环境：JWT保存在客户端，在分布式环境下不需要做额外工作。而session因为保存在服务端，分布式环境下需要实现多机数据共享
  2. 通用性：session一般需要结合Cookie实现认证，所以需要浏览器支持cookie，因此移动端无法使用session认证方案。
  3. 安全性：JWT使用Base64编码，不能存储敏感数据（如密码）；session的信息是存在服务端的，相对来说更安全。
  4. 性能：编码的JWT比较大，cookie放不下，就要放到local storage中，而每次请求token放到Header里面；而sessionId只是很短的一个字符串，因此使用JWT的HTTP请求比使用session的开销大得多。

6. 密码是如何存储的？

  1. 使用HTTPS协议，数据直接加密传输，无需额外加密；
  2. 密码传输时客户端用base64编码传输，服务端用base64解码得到原密码。
  3. 密码入库时使用PBKDF2（PBKDF2WithHmacSHA1）算法。
  4. 登录时，将输入密码与数据库中的密码以同样的方式加密之后进行比对。

JavaScript部分

  1. var、let、const区别？

    • let和const是ES6新增的声明变量的关键词，之前声明变量的关键词是var。

    • 变量提升：var声明的变量存在变量提升，即变量可以在声明之前调用，值为undefined ；let和const不存在变量提升，即它们所声明的变量一定要在声明后使用，否则报错。

    • 块级作用域：var不存在块级作用域；let和const存在块级作用域。

    • 重复声明：var允许重复声明变量；let和const在同一作用域不允许重复声明变量。

    • 修改声明的变量：var和let可以；const声明一个只读的常量。一旦声明，常量的值就不能改变。

    • 使用：能用const的情况尽量使用const，其他情况下大多数使用let，避免使用var。

  2. ==和===区别？

    • 相等操作符（==）(值相等就相等)会做类型转换，再进行值的比较，全等运算符(===)(值和引用地址都相等才相等)不会做类型转换；
    • null 和 undefined 比较，相等操作符（==）为true，全等为false。

  3. 本地存储的方式有哪些？区别及应用场景？

    • 本地缓存有：cookie、sessionStorage、localStorage
    • 区别：
      • 存储大小：cookie数据大小不能超过4k，sessionStorage和localStorage虽然也有存储大小的限制，但比cookie大得多，可以达到5M或更大。
      • 有效时间：localStorage存储持久数据，浏览器关闭后数据不丢失除非主动删除数据； sessionStorage数据在当前浏览器窗口关闭后自动删除；cookie设置的cookie过期时间之前一直有效，即使窗口或浏览器关闭。
      • 数据与服务器之间的交互方式：cookie的数据会自动的传递到服务器，服务器端也可以写cookie到客户端； sessionStorage和localStorage不会自动把数据发给服务器，仅在本地保存。
    • 应用场景
      • 标记用户与跟踪用户行为（用户登录信息）的情况，推荐使用cookie
      • 适合长期保存在本地的数据（令牌），推荐使用localStorage
      • 敏感账号一次性登录，推荐使用sessionStorage

  4. 对闭包的理解

    • 闭包就是函数中包含另一个函数，可以让你在函数外部读取到内部的变量（就是在函数内部再定义一个函数），让这些变量的值始终保持在内存中，可以达到延长变量生命周期的效果，过多使用会导致内存泄漏的问题。
    • 在创建私有变量和想延长变量的生命周期时会用到闭包。

  5. 深拷贝浅拷贝的区别？

    • 浅拷贝：如果属性是基本类型，拷贝的就是基本类型的值。如果属性是引用类型，拷贝的就是内存地址
    • 深拷贝：深拷贝开辟一个新的栈，两个对象属完成相同，但是对应两个不同的地址，修改一个对象的属性，不会改变另一个对象的属性

  6. JS中的数据类型？

    • string、number、boolean、undefined、null、object

1. ROS的通信机制？

  • ROS基本的通信方式有三种：话题通信（发布订阅模式）、服务通信（请求响应模式）和参数服务器（参数共享模式）。
  • 话题通信指的是一个节点发布消息，另一个节点订阅消息，适用于不断更新的、少逻辑处理的数据传输场景。话题通信涉及到Talker（发布者）、Listener（订阅者）和ROS Master（管理者）三个角色。
  • 服务通信是基于请求响应模式的，是一种应答机制。也即: 一个节点A向另一个节点B发送请求，B接收处理请求并产生响应结果返回给A。一般用于偶然的、对时时性有要求、有一定逻辑处理需求的数据传输场景。服务通信涉及到Client（客户端）、Server（服务端）和ROS Master（管理者）三个角色。

2. ROS的常用命令？

  • catkin_creare_pkg：创建包
  • catkin_make：编译ros空间
  • roscore：启动Ros master
  • rosrun：运行ros包中的一个可执行文件。rosrun只能运行一个节点，如果要运行多个节点，就需要多次使用rosrun命令。
  • roslaunch：启动roscore，本地节点，远程节点（通过ssh），设置参数服务器参数。roslaunch可以采用xml格式描述运行的节点，同时运行多个节点。同时，通过roslaunch启动节点时，不必提前单独开启roscore。
  • rostopic：获取topic的相关信息， 包括发布者、订阅者、发布频率和messages。
  • rosparam：获取和设定ros参数服务器（Parameter Server）上的数据。

3. SLAM是什么？

SLAM问题可以描述为: 机器人在未知环境中从一个未知位置开始移动,在移动过程中根据位置估计和地图进行自身定位，同时在自身定位的基础上建造增量式地图，以绘制出外部环境的完全地图。

4. 什么是重定位？

重定位就是重新定位，当前状态的机器人无法确定其在地图中的位置，也叫做机器人被“绑架”，就说的是人质被蒙上双眼带到未知地方，蒙罩去掉后完全不知道自己在哪里，这时候就需要充分利用之前建好的地图或者存好的数据库。此时机器人需要观察周围环境，并且从已有地图中寻找可靠的匹配关系，一般是关键帧信息，这样就可以根据已有信息“重新”估计机器人的姿态。
